# 06. Frontend Q&A Response List

이 문서는 프론트엔드 발표 이후 예상 질문에 대응하기 위한 답변 리스트입니다. 답변은 발표장에서 바로 말할 수 있도록 짧게 작성하되, 과장하면 안 되는 부분은 한계와 개선 방향까지 함께 적었습니다.

## 답변 원칙

- 모르는 부분은 구현된 범위와 개선 예정 범위를 분리해서 답한다.
- "완벽히 보안 처리했다"처럼 과장하지 않는다. 클라이언트 검증은 1차 방어이고 최종 보안은 백엔드 검증이 필요하다고 답한다.
- S3 업로드는 현재 시뮬레이션이다. 실제 분석 요청은 base64 이미지를 백엔드로 보내는 구조라고 정확히 말한다.
- `X-Session-ID`는 인증 수단이 아니라 익명 요청 추적용 correlation ID라고 답한다.
- shadcn/Radix는 기반 전략이고, 현재 화면 대부분은 프로젝트 소유 커스텀 컴포넌트라고 답한다.

## 1. 프로젝트 전체

Q. 프론트엔드가 이 프로젝트에서 맡은 핵심 역할은 무엇인가요?

A. 사용자의 이미지와 향기 취향을 입력받고, 백엔드 분석 결과를 이해 가능한 리포트와 추천 UI로 바꾸는 역할입니다. 단순 화면이 아니라 입력, 분석 요청, 상태 관리, 시각화, 공유까지 사용자 경험 전체를 담당합니다.

Q. 사용자는 어떤 흐름으로 서비스를 이용하나요?

A. 개인정보 동의 후 향기 노트를 선택하고, OOTD 이미지를 업로드한 뒤 분석을 시작합니다. 이후 백엔드 결과를 기반으로 향기 아우라 리포트와 추천 향수 리스트, 상세 모달을 확인합니다.

Q. 프론트엔드에서 가장 신경 쓴 부분은 무엇인가요?

A. 이미지 업로드와 분석 요청이 중복 실행되지 않게 하는 안정성, TypeScript 기반 데이터 계약, Tailwind 기반 브랜드 UI 일관성, 리포트 캡처 품질입니다.

Q. 단순한 랜딩 페이지와 무엇이 다른가요?

A. 랜딩 페이지처럼 스크롤형 구성이지만, 실제로는 이미지 업로드, API 요청, 전역 상태, 추천 정렬, 리포트 저장, 상세 모달이 연결된 인터랙티브 애플리케이션입니다.

Q. 프론트엔드가 AI 분석을 직접 하나요?

A. AI 분석 자체는 백엔드가 담당합니다. 프론트엔드는 이미지를 검증 및 경량화하고, 선택 노트와 함께 API로 보내며, 응답 결과를 시각화합니다.

Q. 백엔드가 없으면 화면이 동작하나요?

A. 분석 API는 백엔드가 필요합니다. 다만 추천 결과가 없을 때를 대비해 프론트엔드에 fallback 추천 로직과 로컬 상품 데이터가 있어 리포트 UI가 완전히 비지 않도록 설계했습니다.

Q. 이 프로젝트에서 프론트엔드의 기술적 난이도는 어디에 있나요?

A. 이미지 파일 처리, 비동기 분석 상태, 중복 요청 방지, 백엔드 응답 타입 관리, 리포트 캡처 품질 보정, 반응형 UI를 동시에 다뤄야 했다는 점입니다.

Q. 프론트엔드에서 데이터 흐름을 한 문장으로 설명하면?

A. 사용자 입력을 Zustand 상태와 API 요청으로 연결하고, 백엔드 결과를 리포트와 추천 컴포넌트가 사용할 수 있는 파생 데이터로 변환하는 흐름입니다.

## 2. 기술 스택 선택

Q. 왜 TypeScript를 선택했나요?

A. 백엔드 응답, 추천 상품, 리포트 데이터처럼 여러 컴포넌트가 공유하는 객체 구조가 많아서 타입으로 계약을 고정할 필요가 있었습니다. 덕분에 필드 누락이나 잘못된 props 연결을 빌드 단계에서 잡을 수 있습니다.

Q. JavaScript로도 구현 가능한데 TypeScript가 꼭 필요했나요?

A. 기능 자체는 JavaScript로도 만들 수 있습니다. 하지만 팀 프로젝트에서는 데이터 구조가 자주 바뀌고 여러 명이 같은 타입을 공유하므로, TypeScript가 협업 안정성과 리팩토링 안전성을 높여 줍니다.

Q. TypeScript를 쓰면 런타임 오류가 없어지나요?

A. 아닙니다. TypeScript는 개발 단계의 타입 오류를 줄이는 도구입니다. 서버에서 예상과 다른 JSON이 내려오는 문제는 API 계약, 테스트, 필요하면 런타임 스키마 검증으로 보완해야 합니다.

Q. 왜 React를 선택했나요?

A. 화면을 섹션과 컴포넌트 단위로 나눠 관리하기 좋고, 상태 변화에 따라 UI를 다시 그리는 방식이 분석 결과 리포트처럼 데이터 기반 화면에 적합했기 때문입니다.

Q. Vue나 Svelte 대신 React를 쓴 이유는?

A. 팀 내 학습 자료와 생태계, 라이브러리 지원, Vite와의 조합, TypeScript 지원을 고려했습니다. 프로젝트에서 필요한 상태 관리, 차트, 캡처, 테스트 도구를 안정적으로 조합하기 쉬웠습니다.

Q. 왜 Next.js가 아니라 Vite인가요?

A. 이번 서비스는 검색 엔진 최적화나 서버 렌더링보다 이미지 업로드, 분석 요청, 리포트 인터랙션이 중심입니다. 그래서 빠른 개발 서버와 가벼운 CSR 구성이 장점인 Vite가 적합했습니다.

Q. Vite의 장점은 무엇인가요?

A. 개발 서버 시작과 HMR이 빠르고, React + TypeScript 설정이 단순합니다. 빌드 도구 설정이 과하지 않아 팀 프로젝트에서 개발 속도를 높일 수 있었습니다.

Q. Tailwind CSS를 선택한 이유는?

A. 컴포넌트 안에서 스타일을 바로 조합할 수 있고, 브랜드 색상과 간격을 설정 파일로 통일할 수 있습니다. 여러 화면을 빠르게 만들면서도 UI 일관성을 유지하기 좋았습니다.

Q. 일반 CSS와 Tailwind CSS의 핵심 차이는?

A. 일반 CSS는 클래스 이름을 직접 만들고 별도 파일에 스타일을 작성합니다. Tailwind는 `px-6`, `bg-wood`, `md:py-40`처럼 작은 유틸리티 클래스를 조합해 스타일을 만듭니다.

Q. Tailwind를 쓰면 CSS를 몰라도 되나요?

A. 아닙니다. Tailwind는 CSS 속성을 유틸리티 클래스로 표준화한 도구입니다. flex, grid, spacing, responsive, transition 같은 CSS 개념을 이해해야 제대로 사용할 수 있습니다.

Q. Tailwind className이 너무 길어지지 않나요?

A. 길어질 수 있습니다. 그래서 반복되는 UI는 컴포넌트로 분리하고, 전역적으로 반복되는 패턴은 `index.css`의 `@layer components`에 공통 클래스로 정리했습니다.

Q. styled-components 대신 Tailwind를 쓴 이유는?

A. 이번 프로젝트는 디자인 토큰을 반복 조합하는 화면이 많았습니다. Tailwind는 별도 스타일 파일이나 런타임 스타일 생성 없이 빠르게 반응형 UI를 만들 수 있어 더 적합했습니다.

Q. CSS Module은 고려하지 않았나요?

A. CSS Module도 좋은 선택이지만, 이 프로젝트는 컴포넌트 단위로 빠르게 레이아웃과 상태 스타일을 조정하는 일이 많았습니다. Tailwind가 개발 속도와 디자인 일관성 측면에서 더 맞았습니다.

Q. Zustand를 선택한 이유는?

A. 전역 상태 범위가 동의 여부, 선택 노트, 분석 결과, 모달, 로딩/에러 정도로 명확했습니다. Redux보다 boilerplate가 적고 훅처럼 쓸 수 있어 프로젝트 규모에 맞았습니다.

Q. Redux를 쓰지 않은 이유는?

A. Redux는 큰 규모의 복잡한 상태 변경 이력 관리에 강점이 있지만, 이번 프로젝트에는 설정과 코드량이 과할 수 있습니다. Zustand가 필요한 전역 상태를 더 단순하게 표현할 수 있었습니다.

Q. React Context로도 가능한 것 아닌가요?

A. 가능합니다. 다만 Context는 값이 바뀔 때 관련 컴포넌트 리렌더링 관리가 번거로워질 수 있습니다. Zustand는 store 기반으로 액션과 상태를 분리하기 쉬워 선택했습니다.

Q. Axios를 선택한 이유는?

A. 요청 인스턴스, timeout, request/response interceptor를 한곳에서 관리하기 쉽기 때문입니다. 세션 헤더 주입과 공통 에러 처리를 API 서비스 계층에 모을 수 있었습니다.

Q. fetch 대신 Axios를 써야 했나요?

A. fetch로도 구현할 수 있습니다. 다만 Axios는 JSON 처리, timeout, interceptor 패턴이 편해서 공통 API 계층을 빠르게 구성하기 좋았습니다.

Q. Recharts를 쓰나요?

A. 의존성에는 Recharts가 있지만 현재 핵심 레이더 차트는 `RadarChart.tsx`에서 SVG 기반 커스텀 구현을 사용합니다. 직접 제어가 필요한 애니메이션과 툴팁 표현에 맞췄습니다.

Q. html2canvas를 왜 사용했나요?

A. 리포트 DOM을 사용자가 저장할 수 있는 PNG 이미지로 변환하기 위해 사용했습니다. 단순 스크린샷이 아니라 리포트 영역만 캡처하는 기능이 필요했습니다.

Q. Lucide React는 왜 썼나요?

A. 아이콘 스타일이 일관되고 React 컴포넌트로 바로 사용할 수 있습니다. 업로드, 삭제, 재시작, 닫기 같은 UI 액션을 텍스트만으로 표현하는 것보다 직관적입니다.

## 3. React 구조와 컴포넌트

Q. `App.tsx`의 역할은 무엇인가요?

A. 전체 섹션 조립, 개인정보 동의 흐름, 전역 상태 연결, 분석 결과 전달, 상품 모달 렌더링을 담당하는 최상위 컨테이너입니다.

Q. 섹션을 왜 나눴나요?

A. Hero, 향기 가이드, AI 인터뷰, 리포트, 안전 가치처럼 화면 목적이 다르기 때문입니다. 섹션 단위로 나누면 유지보수와 lazy loading, 에러 격리가 쉬워집니다.

Q. lazy loading을 왜 적용했나요?

A. 초기 화면에서 모든 섹션 코드를 한 번에 불러오지 않기 위해서입니다. 사용자가 스크롤하면서 보게 되는 섹션을 지연 로딩하면 초기 로딩 부담을 줄일 수 있습니다.

Q. `Suspense`는 왜 쓰나요?

A. lazy loading 중인 컴포넌트의 렌더링 경계를 잡기 위해 사용합니다. 현재 구현은 Skeleton 컴포넌트를 삭제했고, `Suspense fallback={null}`로 섹션 로딩 중 별도 UI를 표시하지 않습니다.

Q. `ErrorBoundary`는 왜 쓰나요?

A. 특정 섹션에서 렌더링 오류가 발생해도 전체 앱이 멈추지 않게 하기 위해서입니다. 섹션 단위로 장애 범위를 줄이는 목적입니다.

Q. SPA 구조인가요?

A. 네. 메인 플로우는 라우터 이동보다 단일 페이지 스크롤 흐름에 가깝습니다. 사용자는 같은 페이지에서 입력, 분석, 리포트 확인까지 진행합니다.

Q. React Router가 있는데 왜 라우팅을 적극적으로 쓰지 않았나요?

A. 현재 핵심 UX는 섹션 간 스크롤 이동입니다. 별도 URL 페이지로 나누기보다 사용자가 한 흐름에서 서비스를 경험하게 하는 것이 더 자연스러웠습니다.

Q. props drilling 문제가 있지 않나요?

A. 현재는 최상위에서 핵심 상태를 받아 일부 섹션에 props로 전달하는 구조입니다. 상태 범위가 크지 않아 관리 가능하지만, 규모가 커지면 selector 기반으로 각 컴포넌트가 필요한 상태만 구독하도록 개선할 수 있습니다.

Q. `restartToken`은 왜 있나요?

A. 재분석을 시작할 때 특정 섹션을 초기 상태로 다시 마운트하기 위한 key 값입니다. React의 key 변경을 이용해 내부 상태를 초기화합니다.

Q. `ProductModal`을 main 콘텐츠 밖에 둔 이유는?

A. 메인 콘텐츠에 blur나 transform이 적용될 수 있기 때문에, 모달은 그 영향을 받지 않도록 독립 레이어로 배치했습니다.

Q. 컴포넌트 폴더를 어떻게 나눴나요?

A. `sections`는 페이지 큰 단위, `common`은 공통 UI, `report`는 리포트 전용, `guide`는 향기 가이드 전용, `curated`는 상품 상세 모달처럼 도메인별로 나눴습니다.

Q. 절대 경로 alias `@`는 왜 쓰나요?

A. `../../../components` 같은 긴 상대 경로를 줄이고 파일 이동 시 import 경로를 관리하기 쉽게 하기 위해서입니다.

Q. `useIntersectionObserver`는 왜 만들었나요?

A. 스크롤로 요소가 화면에 진입했을 때 한 번만 애니메이션을 실행하기 위해 만들었습니다. AOS 같은 외부 라이브러리 대신 필요한 기능만 직접 구현했습니다.

## 4. 상태 관리

Q. 전역 상태에는 무엇이 들어가나요?

A. 개인정보 동의 여부, 선택한 향기 노트, 분석 결과, 선택된 상품, 로딩 상태, 에러 상태, 재시작 토큰이 들어갑니다.

Q. 개인정보 동의 상태는 어디에 저장하나요?

A. Zustand store에서 관리하고, 새로고침 후에도 유지되도록 `localStorage`의 `olfit_consent`, `olfit_session_id`와 연결합니다.

Q. localStorage를 쓰는 이유는?

A. 사용자가 새로고침할 때마다 동의 모달이 다시 뜨지 않도록 하기 위해서입니다. 세션 ID도 익명 요청 추적을 위해 보관합니다.

Q. localStorage는 안전한가요?

A. 민감한 인증 정보 저장소로는 적합하지 않습니다. 현재 저장하는 값은 동의 여부와 익명 세션 ID이며, 인증이나 권한 판단에는 사용하면 안 됩니다.

Q. `X-Session-ID`는 보안 토큰인가요?

A. 아닙니다. 프론트에서 생성한 값이라 사용자가 조작할 수 있습니다. 백엔드에서는 인증이 아니라 로그 추적이나 요청 묶기 정도의 correlation ID로 취급해야 합니다.

Q. 로딩 상태는 어디서 관리하나요?

A. Zustand의 `isLoading`으로 전역 관리합니다. `requestAuraAnalysis()`가 시작될 때 true, 끝날 때 false로 바꿉니다.

Q. 에러는 어떻게 처리하나요?

A. Axios response interceptor에서 공통 에러 메시지를 Zustand의 `error`에 저장하고, UI에서는 `ErrorFallback` 같은 컴포넌트로 사용자에게 안내합니다.

Q. `selectedProduct`는 왜 전역 상태인가요?

A. 추천 리스트에서 선택한 상품을 최상위 모달로 띄워야 하기 때문입니다. 리스트 컴포넌트와 모달 렌더링 위치가 달라 전역 또는 상위 상태가 필요합니다.

Q. `analysisResults`가 null일 때는 어떻게 하나요?

A. 리포트 섹션에서 분석 대기 상태 UI를 보여줍니다. 분석이 완료되면 결과를 기반으로 리포트와 추천을 렌더링합니다.

Q. `resetAll()`은 무엇을 하나요?

A. 분석 결과, 선택 노트, 선택 상품, 로딩/에러를 초기화하고 `restartToken`을 증가시켜 재분석 플로우를 초기화합니다.

## 5. API와 데이터 계약

Q. 분석 API 요청에는 어떤 데이터가 들어가나요?

A. base64 이미지와 사용자가 선택한 향기 노트 배열이 들어갑니다. 요청 함수는 `requestAuraAnalysis(base64Image, selectedNotes)`입니다.

Q. API 기본 주소는 어디서 정하나요?

A. `import.meta.env.VITE_API_URL`을 우선 사용하고, 없으면 `http://localhost:8000`을 기본값으로 사용합니다.

Q. API timeout은 왜 30초인가요?

A. 이미지 분석은 일반 텍스트 요청보다 시간이 걸릴 수 있습니다. 너무 짧으면 정상 분석도 실패할 수 있어 30초 timeout을 설정했습니다.

Q. 백엔드 추천 결과가 있으면 어떻게 하나요?

A. `results.recommendations`가 있으면 이를 우선 사용합니다. 프론트엔드 로컬 scoring은 백엔드 추천이 없을 때만 fallback으로 사용합니다.

Q. Product 타입은 어디에서 쓰나요?

A. 추천 리스트, 상품 카드, 상세 모달, 분석 결과의 recommendations에서 공통으로 사용합니다. 이름, 브랜드, 가격, 이미지, 노트, 상세 설명을 포함합니다.

Q. `price`와 `price_krw`를 둘 다 둔 이유는?

A. `price`는 화면 표시용 문자열이고, `price_krw`는 정렬과 계산용 숫자입니다. 표시용 문자열을 파싱하면 통화와 포맷 문제로 오류가 생길 수 있어 분리했습니다.

Q. `radarScores`는 무엇인가요?

A. 백엔드가 계산한 5축 향기 성향 점수입니다. 프론트엔드는 이 값을 레이더 차트 데이터로 변환해 시각화합니다.

Q. 백엔드 응답 구조가 바뀌면 어떻게 대응하나요?

A. 공유 타입과 API 계약을 먼저 수정하고, 리포트 파생 로직인 `useInsightReport.ts`에서 UI가 필요한 형태로 변환합니다. 변경 범위를 한곳에 모으는 것이 중요합니다.

Q. API 에러가 나면 추천 결과는 어떻게 되나요?

A. 분석 요청 자체가 실패하면 에러 UI와 재시도 안내를 보여줍니다. 추천 fallback은 분석 결과 객체가 있을 때 사용하는 구조라, API 실패를 완전히 대체하지는 않습니다.

Q. 프론트엔드에서 추천 로직을 왜 가지고 있나요?

A. 백엔드 추천이 없거나 개발 중인 상황에서도 UI가 동작하게 하기 위한 fallback입니다. 운영 기준으로는 백엔드 추천 결과를 우선합니다.

Q. 추천 사유는 어떻게 만드나요?

A. 로컬 fallback에서는 매칭된 키워드나 선택 노트를 기반으로 `matchReason` 문장을 생성합니다. 백엔드 추천이 있으면 백엔드가 내려준 추천 사유를 우선합니다.

## 6. 이미지 업로드와 보안

Q. 이미지 업로드에서 어떤 검증을 하나요?

A. MIME 타입, 확장자, magic number, 파일 크기, 이미지 해상도, 이미지 decode 가능 여부를 확인합니다.

Q. magic number 검사는 왜 하나요?

A. 확장자만 이미지로 바꾼 악성 파일을 1차적으로 걸러내기 위해서입니다. 파일 앞부분의 바이너리 시그니처가 JPEG, PNG, WEBP 형식인지 확인합니다.

Q. MIME 타입과 확장자를 둘 다 검사하는 이유는?

A. 둘 중 하나만으로는 위조 가능성이 있습니다. 둘 다 확인하면 단순한 잘못된 업로드나 위장 파일을 더 잘 걸러낼 수 있습니다.

Q. 파일 크기 제한은 왜 10MB인가요?

A. 너무 큰 이미지는 업로드와 base64 변환, canvas 처리에서 성능 문제를 일으킬 수 있습니다. 사용자 경험과 DoS성 입력 방지를 위해 제한했습니다.

Q. 해상도 4096px 제한은 왜 있나요?

A. 픽셀 수가 큰 이미지는 브라우저 메모리를 많이 사용합니다. 이미지 폭이나 높이가 지나치게 큰 경우 canvas 처리 전에 차단합니다.

Q. 이미지를 왜 1200px로 리사이징하나요?

A. AI 분석과 미리보기에는 원본 초고해상도가 필요하지 않은 경우가 많습니다. 크기를 줄이면 전송량과 처리 비용을 낮출 수 있습니다.

Q. base64로 변환하는 이유는?

A. 현재 분석 API가 이미지 데이터를 JSON으로 받는 구조이기 때문입니다. canvas로 리사이징한 뒤 JPEG base64로 변환해 API에 전달합니다.

Q. base64 방식의 단점은?

A. 바이너리보다 데이터 크기가 커지고, 큰 이미지를 메모리에 올려야 합니다. 실제 운영에서는 스토리지 업로드 후 URL이나 파일 토큰을 보내는 방식도 고려할 수 있습니다.

Q. 현재 이미지는 실제 S3에 올라가나요?

A. 아닙니다. `uploadToCloudStorage()`는 업로드 과정을 시뮬레이션하고 fake URL을 반환합니다. 실제 분석 요청은 base64 이미지를 백엔드로 보내는 구조입니다.

Q. 그런데 UI에 S3 업로드라고 보이는 이유는?

A. 향후 실제 스토리지 연동을 고려한 UX 흐름을 미리 반영한 것입니다. 발표에서는 현재 구현이 시뮬레이션이라는 점을 명확히 말해야 합니다.

Q. 업로드 중복 요청은 어떻게 막았나요?

A. `ImageUploader`에서 `uploadProcessingRef`로 처리 중 여부를 즉시 잠그고, `AIInterviewSection`에서도 `processingRef`로 분석 중복 실행을 한 번 더 막습니다.

Q. 왜 state가 아니라 ref로 중복 요청을 막나요?

A. React state는 업데이트가 비동기로 반영됩니다. 아주 빠른 double drop이나 클릭에서는 state 반영 전에 함수가 다시 들어올 수 있어, 즉시 값이 바뀌는 ref lock을 사용했습니다.

Q. 이미지 선택 후 바로 분석하지 않고 버튼을 누르게 한 이유는?

A. 사용자가 업로드된 이미지를 미리 확인하고, 원하지 않으면 초기화할 수 있도록 하기 위해서입니다. 분석 요청을 명시적인 사용자 액션으로 분리했습니다.

Q. drag & drop도 지원하나요?

A. 네. 업로드 영역에 `onDragOver`, `onDragLeave`, `onDrop`을 구현해 파일 선택과 드래그 앤 드롭을 모두 지원합니다.

Q. `accept="image/*"`만으로 충분하지 않나요?

A. 충분하지 않습니다. accept는 브라우저 파일 선택 UI의 힌트에 가깝습니다. 실제 검증은 코드와 백엔드에서 별도로 해야 합니다.

Q. 클라이언트 검증만으로 보안이 완성되나요?

A. 아닙니다. 클라이언트 검증은 UX와 1차 방어입니다. 백엔드에서도 파일 타입, 크기, 내용, 저장 정책을 반드시 다시 검증해야 합니다.

Q. 이미지 개인정보는 어떻게 다루나요?

A. 프론트엔드는 분석 전 개인정보 동의를 받고, 이미지를 리사이징해 전송합니다. 다만 저장, 보관 기간, 삭제 정책 같은 최종 개인정보 처리는 백엔드와 운영 정책에서 명확히 해야 합니다.

## 7. 분석 진행 UX

Q. 분석 진행률은 실제 서버 진행률인가요?

A. 아닙니다. 현재 progress는 사용자가 기다리는 동안 분석 단계가 진행 중임을 보여주는 UX용 진행률입니다. 실제 서버 진행률과 1:1로 연결된 값은 아닙니다.

Q. 발표자가 실제 화면에서 분석 진행률을 못 봤다면 왜 그런가요?

A. 이 UI는 업로드 직후 자동으로 뜨는 화면이 아니라, 이미지 미리보기 후 `분석 시작` 버튼을 눌렀을 때 `isLoading` 동안 조건부로 표시됩니다. 시연에서 못 봤다면 분석 시작 전 상태였거나, 진행 구간이 빠르게 지나갔거나, 에러 상태로 바로 전환된 경우일 수 있습니다.

Q. 왜 가짜 progress를 쓰나요?

A. 이미지 분석 요청은 몇 초 동안 화면 변화가 없을 수 있습니다. 단계 문구와 progress를 제공하면 사용자가 서비스가 멈춘 것이 아니라 처리 중이라고 이해할 수 있습니다.

Q. 분석 단계 문구는 어떤 의미인가요?

A. 이미지 픽셀 데이터 추출, 스타일 분석, 색채 심리 기반 무드 매칭, 선택 노트 결합 같은 사용자가 이해하기 쉬운 처리 흐름을 설명하는 UX 문구입니다.

Q. API 요청은 언제 실행되나요?

A. 이미지 업로드와 미리보기 확인 후 사용자가 `분석 시작`을 누르면 진행률 UI가 표시되고, 현재 구현에서는 약 5초 뒤 `requestAuraAnalysis()`를 1회 호출합니다.

Q. 왜 API를 바로 호출하지 않고 약 5초 뒤에 호출하나요?

A. 현재 구현은 데모에서 분석 과정이 단계적으로 진행되는 것처럼 보여주기 위한 UX 시뮬레이션이 먼저 들어가 있습니다. 다만 운영 기준으로는 `분석 시작` 직후 API를 바로 호출하고, 서버 응답을 기다리는 동안 progress나 loading indicator를 보여주는 구조가 더 자연스럽습니다.

Q. 분석 100% 이후 랜덤 향기 팁을 보여주나요?

A. 아닙니다. 현재 코드에는 `scentTips`나 `randomTip` 기반 랜덤 팁 기능이 없습니다. 100% 이후에는 `getFinalizingSteps()`의 고정 문구를 순환 표시합니다.

Q. 100% 이후 어떤 문구가 표시되나요?

A. `아우라 분석 결과를 정리하는 중...`, `추천 향수 데이터를 불러오는 중...`, `제품 이미지와 상세 정보를 준비하는 중...`, `리포트 화면을 구성하는 중...` 같은 마무리 상태 문구가 순환됩니다.

Q. 분석 실패 시 재시도는 어떻게 하나요?

A. 에러 상태가 표시되고 `ErrorFallback`을 통해 재시도할 수 있습니다. 실패한 경우에는 분석 lock을 해제해 다시 요청할 수 있도록 했습니다.

Q. 분석 완료 후 다시 요청할 수 있나요?

A. 완료 상태에서는 같은 요청이 중복 실행되지 않도록 막고, 리포트의 다시하기 버튼으로 전체 플로우를 초기화한 뒤 재분석할 수 있습니다.

## 8. 추천과 리포트

Q. 추천 리스트는 어떻게 만들어지나요?

A. 백엔드 recommendations가 있으면 그대로 사용하고, 없으면 선택 노트와 키워드, 상품 데이터를 기반으로 로컬 fallback 추천을 계산합니다.

Q. 로컬 fallback 점수는 어떻게 계산하나요?

A. 사용자가 선택한 노트와 상품의 노트 정보가 일치하면 점수를 주고, 백엔드 향수 키워드가 상품 family나 story에 매칭되면 추가 점수를 줍니다. 키워드가 없으면 무드 기반 보조 매칭을 사용합니다.

Q. 유사도에 stable random이 들어가는 이유는?

A. 완전히 같은 점수만 나오면 UI가 단조로워질 수 있어 제품 ID 기반의 고정 보정값을 더합니다. 매번 랜덤하게 바뀌는 것이 아니라 같은 제품에는 같은 보정이 적용됩니다.

Q. 이 유사도는 과학적으로 정확한 수치인가요?

A. 프론트엔드 fallback의 유사도는 UI용 보조 점수입니다. 운영 기준의 추천 신뢰도는 백엔드 모델과 데이터 기반 추천 결과를 우선해야 합니다.

Q. 추천순과 가격순은 어떻게 다르나요?

A. 추천순은 백엔드 또는 fallback이 정한 유사도 순서이고, 가격순은 `price_krw` 숫자 값을 기준으로 낮은 가격부터 정렬합니다.

Q. 가격 정보가 없으면 어떻게 정렬되나요?

A. `price_krw`가 없거나 유효하지 않으면 `Infinity`로 처리해 가격 정보가 있는 상품 뒤로 보냅니다.

Q. 최신 UI refinement에서 Safety Values 문구는 어떻게 바뀌었나요?

A. `알레르기`, `에코 비건`, `TPO`처럼 넓게 보일 수 있는 문구 대신 `IFRA 성분 가이드`, `브랜드 윤리성 공개`, `아우라 매칭 알고리즘`으로 실제 데이터와 로직 기준에 맞춰 구체화했습니다.

Q. Scent Family Carousel은 무엇이 개선됐나요?

A. 자동 전환 주기를 30초로 늘렸고, hover 또는 focus 상태에서 좌우 이동 버튼이 보이도록 했습니다. 사용자가 수동 조작하면 `manualResetKey`를 증가시켜 다음 자동 전환 타이머를 다시 잡습니다.

Q. Philosophy Section의 수치는 무엇을 의미하나요?

A. 현재 UI에는 `55,717+` 분석 데이터, `13+` 프리미엄 브랜드, `1,240+` 향수 아카이브가 표시됩니다. 발표에서는 실제 DB/아카이브 기반 신뢰 지표로 설명하되, 산출 기준은 팀 데이터 기준에 맞춰 답하면 됩니다.

Q. 레이더 차트는 무엇을 보여주나요?

A. 플로랄, 우디, 앰버, 프레시, 구르망 같은 향기 성향 축을 0에서 1 사이 값으로 시각화합니다.

Q. 레이더 차트를 직접 SVG로 구현한 이유는?

A. 축, 점, 툴팁, 애니메이션을 세밀하게 제어하기 위해서입니다. 디자인 톤에 맞는 가벼운 커스텀 차트가 필요했습니다.

Q. `ScentBlueprint`는 어떤 역할인가요?

A. 사용자가 선택한 Top, Middle, Base 노트를 향기 설계도처럼 보여주는 리포트 구성 요소입니다.

Q. `ProductModal`에는 어떤 정보가 나오나요?

A. 상품 이미지, 브랜드, 이름, 가격, 태그, 스토리, Top/Mid/Base 노트, 추천 사용 상황이 표시됩니다.

Q. 상세 모달의 조사 보정은 왜 필요한가요?

A. 한국어 문장에서 상품명 뒤 조사가 어색해질 수 있습니다. `normalizeSubjectParticle()`로 상품명에 맞는 문장 표현을 보정합니다.

Q. 리포트 다시하기는 어떻게 동작하나요?

A. Zustand의 `resetAll()`로 상태를 초기화하고, `interview` 섹션으로 스크롤 이동해 사용자가 다시 분석을 시작할 수 있게 합니다.

## 9. 리포트 캡처

Q. 리포트 저장은 어떻게 구현했나요?

A. `html2canvas`로 리포트 DOM을 canvas로 변환하고, 이를 PNG Blob으로 만들어 다운로드합니다.

Q. 그냥 화면 캡처와 무엇이 다른가요?

A. 브라우저 전체 화면이 아니라 리포트 DOM만 캡처합니다. 또한 캡처용 header, style, placeholder, badge 보정 같은 처리를 추가합니다.

Q. 캡처 전에 폰트 로딩을 기다리는 이유는?

A. 폰트가 로딩되기 전에 캡처하면 글자 폭과 줄바꿈이 달라질 수 있습니다. `document.fonts.ready`를 기다려 결과를 안정화합니다.

Q. 이미지 로딩을 기다리는 이유는?

A. 상품 이미지가 로딩되기 전에 캡처하면 빈 영역이 찍힐 수 있습니다. 모든 이미지가 로딩되거나 timeout될 때까지 기다립니다.

Q. 외부 이미지를 placeholder로 바꾸는 이유는?

A. CORS 헤더가 없는 외부 이미지는 canvas 캡처를 오염시키거나 실패하게 만들 수 있습니다. 그런 이미지는 캡처 전용 placeholder로 대체합니다.

Q. 캡처 때 애니메이션을 끄는 이유는?

A. clone DOM에서 애니메이션이 다시 시작되면 중간 상태가 캡처될 수 있습니다. 캡처 결과를 일정하게 만들기 위해 animation과 transition을 비활성화합니다.

Q. 정렬 버튼과 배지를 canvas로 다시 그리는 이유는?

A. html2canvas가 특정 pill UI의 정렬이나 글자 위치를 브라우저와 다르게 캡처할 수 있어, 캡처 품질을 안정화하기 위한 보정입니다.

Q. 다운로드 중복 클릭은 어떻게 막나요?

A. `reportCapture.ts`의 모듈 레벨 `_isSharing` lock과 `useInsightReport.ts`의 `isCapturingRef`로 중복 캡처와 중복 다운로드를 막습니다.

Q. 공유 API를 쓰지 않고 다운로드만 하나요?

A. 현재 구현은 PNG 다운로드 방식입니다. 모바일 Web Share API 등은 향후 확장할 수 있습니다.

## 10. UI/UX 디테일

Q. 개인정보 동의 전 화면을 blur 처리한 이유는?

A. 사용자가 동의 절차를 먼저 인지하게 하고, 동의 전 업로드나 분석 조작을 막기 위해서입니다.

Q. 배경 상호작용을 왜 막나요?

A. 모달이 떠 있는 동안 뒤쪽 화면을 클릭하면 사용자가 현재 상태를 혼동할 수 있습니다. `pointer-events-none`과 스크롤 차단으로 흐름을 명확히 했습니다.

Q. 브랜드 컬러는 어떻게 관리하나요?

A. `tailwind.config.js`에 `cream`, `wood`, `gold`, `beige`, `oak` 등을 정의하고, JSX에서는 `bg-cream`, `text-wood`처럼 재사용합니다.

Q. 한글 줄바꿈은 어떻게 처리했나요?

A. `break-keep`과 `.text-balance`를 사용해 제목과 본문이 어색하게 끊기지 않도록 했습니다.

Q. 왜 둥근 카드보다 각진 스타일이 많나요?

A. 향수 브랜드의 에디토리얼하고 고급스러운 톤을 위해 과한 라운드보다 얇은 border, 낮은 opacity, 절제된 모서리를 사용했습니다.

Q. Skeleton UI는 쓰고 있나요?

A. 현재는 삭제했습니다. lazy section은 `Suspense fallback={null}`이고, 분석 대기 상태는 Skeleton 대신 overlay, progress bar, `LoadingSpinner`, 에러 fallback으로 표현합니다.

Q. FloatingNavButton은 왜 있나요?

A. 단일 페이지 스크롤 구조에서 사용자가 주요 섹션 사이를 빠르게 이동할 수 있도록 돕기 위한 보조 내비게이션입니다.

Q. Navigation과 Footer는 어떤 역할인가요?

A. Navigation은 서비스의 전체 구조를 안내하고, Footer는 페이지 마무리와 브랜드 정보를 제공합니다. 핵심 분석 흐름과 분리된 레이아웃 컴포넌트입니다.

Q. 모달은 접근성을 고려했나요?

A. 닫기 버튼에 `aria-label`을 넣는 등 일부 고려가 있습니다. 다만 완성형 접근성을 위해서는 focus trap, ESC 닫기, 키보드 이동 보완이 추가로 필요합니다.

Q. 클릭 가능한 div는 접근성 문제가 없나요?

A. 일부 클릭 영역은 버튼으로 바꾸거나 `role`, `tabIndex`, 키보드 핸들링을 추가하는 개선 여지가 있습니다. QA 문서에서도 보완 포인트로 잡을 수 있습니다.

Q. 이미지 alt는 어떻게 처리하나요?

A. 미리보기나 상품 이미지에 `alt`를 넣어 기본 대체 텍스트를 제공합니다. 더 구체적인 alt 정책은 접근성 개선 단계에서 보완할 수 있습니다.

## 11. 성능

Q. 초기 로딩 성능을 위해 무엇을 했나요?

A. 주요 섹션을 `lazy()`로 분리하고 `Suspense` fallback을 사용했습니다. 사용자가 당장 보지 않는 섹션 코드를 초기에 모두 불러오지 않도록 했습니다.

Q. 이미지 처리 성능은 어떻게 관리하나요?

A. 큰 이미지는 크기와 해상도를 제한하고, canvas로 1200px 내로 리사이징해 base64로 변환합니다.

Q. `useMemo`는 어디에 쓰나요?

A. `useInsightReport.ts`에서 추천 목록, 향기 슬롯, 레이더 차트 데이터, 동적 분석 문구처럼 결과에서 파생되는 데이터를 계산할 때 사용합니다.

Q. `useCallback`은 왜 쓰나요?

A. 추천 정렬 버튼이나 다시하기 핸들러처럼 하위 컴포넌트에 전달되는 함수를 메모이제이션해 불필요한 리렌더링 가능성을 줄입니다.

Q. `memo`는 어디에 쓰나요?

A. `RecommendationList`를 `memo`로 감싸 추천 리스트 props가 바뀌지 않으면 불필요한 렌더링을 줄일 수 있게 했습니다.

Q. IntersectionObserver는 성능에 어떤 장점이 있나요?

A. scroll 이벤트를 계속 계산하는 것보다 브라우저가 요소의 뷰포트 진입을 효율적으로 감지합니다. 애니메이션 트리거에 적합합니다.

Q. CSS 애니메이션이 성능에 부담을 주지 않나요?

A. 대부분 opacity, transform, transition 중심으로 사용해 비교적 부담이 적습니다. 캡처 시에는 애니메이션을 꺼서 결과 안정성을 확보합니다.

Q. bundle size는 관리하고 있나요?

A. lazy loading으로 섹션 단위 분리를 적용했습니다. 다만 향후에는 bundle analyzer로 큰 의존성, 이미지 리소스, html2canvas 같은 무거운 라이브러리를 더 세밀히 점검할 수 있습니다.

Q. React 19를 쓴 이유는?

A. 현재 package 기준 React 19를 사용합니다. 최신 React 생태계와 Vite 조합으로 개발했으며, 프로젝트에서는 기본적인 컴포넌트와 훅 패턴 위주로 안정적으로 사용하고 있습니다.

## 12. 테스트와 품질

Q. 어떤 테스트 도구를 사용하나요?

A. Vitest로 단위 테스트를, Playwright로 브라우저 E2E 테스트를 사용합니다. TypeScript build와 ESLint도 정적 검증에 활용합니다.

Q. Vitest에서는 무엇을 테스트하나요?

A. 서비스 함수와 훅의 순수 로직을 테스트합니다. 예를 들어 `sortByPriceKrw()`는 가격 정렬 규칙이 맞는지 검증합니다.

Q. Playwright는 왜 필요한가요?

A. 이미지 선택, drag & drop, API 요청 계측 같은 실제 브라우저 이벤트는 단위 테스트만으로 확인하기 어렵습니다. 그래서 E2E 테스트로 사용자 흐름을 검증합니다.

Q. 빌드 검증 명령은 무엇인가요?

A. `cd frontend && yarn build`입니다. TypeScript build와 Vite production build를 함께 확인합니다.

Q. 단위 테스트 명령은 무엇인가요?

A. `cd frontend && yarn test:run`입니다.

Q. E2E 테스트 명령은 무엇인가요?

A. `cd frontend && yarn test:e2e`입니다. 이 명령은 production build 후 Playwright 테스트를 실행합니다.

Q. 테스트가 모든 버그를 막아주나요?

A. 아닙니다. 테스트는 주요 회귀를 줄이는 장치입니다. 이미지 업로드 중복, 가격 정렬, API 실패, loading 상태, 접근성 같은 중요한 케이스를 계속 늘려야 합니다.

Q. 현재 추가하면 좋은 테스트는 무엇인가요?

A. invalid file rejection, duplicate drop 방지, 추천 상품 상세 모달 검증은 현재 E2E에 반영되어 있습니다. 다음으로는 분석 시작 버튼 중복 클릭 방지, API 실패 retry, 리포트 캡처 중복 클릭 방지, 접근성 keyboard navigation 테스트를 추가하면 좋습니다.

Q. lint는 왜 필요한가요?

A. 사용하지 않는 변수, hooks 규칙 위반, import 문제 등을 사전에 잡아 팀 코드 품질을 일정하게 유지하기 위해 필요합니다.

Q. Tailwind 경고가 있으면 문제가 되나요?

A. 빌드를 막지는 않을 수 있지만 경고가 쌓이면 중요한 경고를 놓칠 수 있습니다. arbitrary value 경고는 named theme utility로 옮기거나 Tailwind 권장 표기로 정리하는 것이 좋습니다.

## 13. 접근성

Q. 접근성에서 신경 쓴 부분은?

A. 버튼, alt, aria-label, 시각적 상태 표시, 명확한 에러 메시지 등을 일부 반영했습니다. 다만 모든 클릭 영역을 키보드 접근 가능하게 만드는 개선은 추가로 필요합니다.

Q. 키보드만으로 사용할 수 있나요?

A. 기본 버튼들은 가능하지만, 일부 카드형 클릭 영역은 추가 개선이 필요합니다. 발표에서는 개선 포인트로 솔직히 말하는 것이 좋습니다.

Q. 모달에서 focus trap은 구현되어 있나요?

A. 현재 커스텀 모달 중심이라 완전한 focus trap은 보완 대상입니다. Radix Dialog 같은 primitive를 적용하면 접근성을 더 안정적으로 확보할 수 있습니다.

Q. 색 대비는 고려했나요?

A. cream/wood 계열을 중심으로 대비를 확보하려고 했습니다. 다만 opacity가 낮은 보조 텍스트는 실제 WCAG 기준으로 추가 점검할 필요가 있습니다.

Q. 스크린 리더 대응은 어느 정도인가요?

A. 기본 alt와 aria-label은 일부 적용했지만, 복잡한 리포트 차트나 캐러셀은 더 명확한 대체 텍스트와 키보드 조작 설명이 필요합니다.

## 14. 배포와 환경

Q. 개발 서버 포트는 무엇인가요?

A. Vite dev server는 `vite.config.ts` 기준 3000번 포트를 사용합니다.

Q. API 서버 주소는 어떻게 바꾸나요?

A. `.env`의 `VITE_API_URL`로 설정할 수 있습니다. 없으면 `http://localhost:8000`을 기본값으로 사용합니다.

Q. Docker 환경에서도 동작하나요?

A. 저장소에는 프론트엔드 Dockerfile과 docker-compose 구성이 있습니다. 프론트엔드, 백엔드, DB를 컨테이너로 함께 띄우는 구조를 고려했습니다.

Q. 정적 배포가 가능한가요?

A. Vite production build 결과물은 정적 파일로 배포할 수 있습니다. 다만 API 서버 주소와 CORS 설정이 맞아야 분석 기능이 동작합니다.

Q. `base: './'` 설정은 왜 있나요?

A. 정적 파일 경로를 상대 경로로 잡아 배포 경로가 루트가 아니어도 asset 접근이 깨지지 않게 하기 위한 설정입니다.

## 15. 보안과 개인정보

Q. 클라이언트에 API key가 있나요?

A. 프론트엔드 코드에는 민감한 API key를 두면 안 됩니다. AI API key 같은 비밀 값은 백엔드에서 관리해야 합니다.

Q. 사용자가 localStorage를 조작하면 어떻게 되나요?

A. 조작할 수 있습니다. 그래서 localStorage 값은 편의 상태나 요청 추적용으로만 쓰고, 권한이나 데이터 소유권 판단에 사용하면 안 됩니다.

Q. XSS 위험은 없나요?

A. 현재 핵심 흐름에서 사용자 입력을 직접 HTML로 렌더링하지 않습니다. 다만 일부 정적 콘텐츠에 `dangerouslySetInnerHTML`이 있다면, API나 CMS 데이터로 확장될 때 sanitize가 필요합니다.

Q. 업로드 파일 검증은 충분한가요?

A. 프론트엔드에서 여러 검증을 하지만 충분하다고 보면 안 됩니다. 백엔드에서 동일하거나 더 강한 검증을 반복해야 합니다.

Q. 개인정보 동의만 있으면 이미지 처리가 법적으로 충분한가요?

A. 법적 충분성은 수집 항목, 보관 기간, 삭제 정책, 제3자 제공 여부 등 운영 정책과 백엔드 처리까지 함께 봐야 합니다. 프론트엔드는 동의 UI와 명시적 진입 제어를 담당합니다.

## 16. 한계와 개선 방향

Q. 현재 가장 큰 한계는 무엇인가요?

A. 클라우드 업로드가 실제 구현이 아니라 시뮬레이션이고, 분석 요청은 base64 전송 구조라는 점입니다. 실제 운영에서는 스토리지 업로드와 이미지 보관 정책을 명확히 해야 합니다.

Q. 프론트엔드에서 개선하고 싶은 부분은?

A. 접근성 보완, E2E 테스트 최신화, 실제 스토리지 업로드 계약 정리, API 응답 런타임 검증, bundle 분석, 이미지 최적화가 우선입니다.

Q. 런타임 타입 검증은 왜 아직 없나요?

A. TypeScript 타입은 개발 단계 검증이라 서버 응답을 런타임에서 보장하지 않습니다. 향후에는 Zod 같은 스키마 검증으로 API 응답을 한 번 더 검증할 수 있습니다.

Q. 추천 로직을 프론트에 두는 것이 맞나요?

A. 운영의 핵심 추천은 백엔드가 담당하는 것이 맞습니다. 프론트 로직은 백엔드 추천이 없을 때 UI를 유지하기 위한 fallback과 presentation layer 역할입니다.

Q. 이미지 업로드를 base64에서 바꾼다면 어떻게 바꾸나요?

A. 프론트에서 실제 스토리지에 업로드하고, 백엔드 분석 API에는 이미지 URL이나 파일 토큰만 보내는 방식으로 바꿀 수 있습니다. 이때 signed URL, 만료 시간, 접근 권한 정책이 필요합니다.

Q. 상태 관리가 더 복잡해지면 어떻게 하나요?

A. Zustand selector를 더 적극적으로 사용해 컴포넌트별 구독 범위를 줄이고, 서버 상태는 React Query 같은 도구로 분리하는 방향을 고려할 수 있습니다.

Q. 대용량 상품 데이터가 들어오면 어떻게 하나요?

A. 현재는 추천 상위 리스트를 렌더링하는 수준입니다. 데이터가 커지면 서버 페이지네이션, virtualization, image lazy loading, 캐싱 전략이 필요합니다.

Q. 모바일 앱으로 확장할 수 있나요?

A. 현재는 웹 기반 React/Vite 구조입니다. 모바일 앱으로 확장하려면 Capacitor 같은 wrapper나 React Native 재구현을 검토할 수 있지만, 현재 구현 범위는 웹입니다.

Q. SSR이 필요해질 가능성은 없나요?

A. 마케팅 SEO가 중요해지거나 초기 콘텐츠 노출이 중요해지면 Next.js 같은 SSR/SSG 구조를 검토할 수 있습니다. 현재는 인터랙션 중심이라 CSR이 적합했습니다.

Q. 디자인 시스템을 더 확장한다면?

A. 공통 Button, Modal, Badge, Card, Dialog 등을 Radix primitive와 Tailwind variant로 정리하고, 반복 className을 줄이는 방향이 좋습니다.

Q. 접근성을 더 강화한다면?

A. 클릭 가능한 영역을 native button으로 바꾸고, focus trap, ESC 닫기, aria-live 에러 알림, 차트 대체 텍스트, 키보드 캐러셀 조작을 추가해야 합니다.

## 17. 사소하지만 나올 수 있는 질문

Q. 왜 파일명이 `AIInterviewSection`인가요?

A. 사용자의 이미지와 취향 정보를 받아 AI 분석으로 이어지는 인터뷰 역할을 하는 섹션이라 그렇게 이름 붙였습니다.

Q. `OOTD`가 무엇인가요?

A. Outfit Of The Day의 약자로, 사용자의 착장 사진을 의미합니다. 이 이미지를 기반으로 스타일 무드를 분석합니다.

Q. 왜 향기 노트를 직접 선택하게 하나요?

A. 이미지만으로는 사용자의 후각 취향을 완전히 알기 어렵습니다. 명시적 선택 노트를 함께 받아 추천 정확도와 설명 가능성을 높입니다.

Q. Top, Middle, Base 노트는 무엇인가요?

A. 향수가 시간에 따라 느껴지는 구조입니다. Top은 첫 향, Middle은 중심 향, Base는 오래 남는 잔향입니다.

Q. 왜 상품 상세에 스토리를 넣나요?

A. 향수 추천은 기능적 매칭뿐 아니라 감성적 설득이 중요합니다. 스토리는 사용자가 추천 이유를 더 잘 이해하게 돕습니다.

Q. 왜 추천 사유를 따로 보여주나요?

A. 단순히 상품만 나열하면 사용자가 왜 추천됐는지 알기 어렵습니다. `matchReason`은 추천의 설명 가능성을 높이는 역할입니다.

Q. 왜 가격순 기능을 넣었나요?

A. 추천 결과를 사용자가 실제 구매 관점에서 비교할 수 있게 하기 위해서입니다. 감성 추천 이후 현실적인 선택 기준을 제공합니다.

Q. 왜 리포트를 이미지로 저장하게 했나요?

A. 사용자가 자신의 분석 결과를 공유하거나 보관할 수 있게 하기 위해서입니다. 서비스 경험을 한 번 보고 끝내지 않게 만드는 기능입니다.

Q. 왜 이미지 미리보기가 필요한가요?

A. 사용자가 잘못된 사진을 올렸는지 확인하고 분석 전에 취소할 수 있어야 합니다. 분석 요청 비용과 사용자 실수를 줄입니다.

Q. 왜 분석 시작 버튼을 비활성화하나요?

A. 업로드가 끝나기 전이나 처리된 이미지가 없을 때 분석 요청이 나가지 않게 하기 위해서입니다.

Q. 왜 `rounded-sm`을 많이 쓰나요?

A. Olfit의 디자인 톤은 둥글고 캐주얼한 앱보다 에디토리얼하고 절제된 브랜드 느낌에 가깝기 때문입니다.

Q. 왜 `cream` 배경을 쓰나요?

A. 향수/뷰티 서비스의 부드럽고 프리미엄한 느낌을 주고, `wood` 텍스트와 대비가 잘 맞기 때문입니다.

Q. 왜 `wood` 컬러를 쓰나요?

A. 검정색보다 부드럽고 브랜드 무드에 맞는 깊은 갈색 계열이라 텍스트와 포인트 색으로 사용했습니다.

Q. 왜 스크롤 애니메이션을 직접 만들었나요?

A. 필요한 기능은 요소가 보일 때 한 번 등장하는 정도라 외부 라이브러리보다 `IntersectionObserver` 훅으로 가볍게 구현했습니다.

Q. 왜 report 로직을 hook으로 뺐나요?

A. `InsightReportSection`에 계산, 정렬, 캡처, visibility 상태가 모두 들어가면 너무 복잡해집니다. `useInsightReport`로 비즈니스 로직을 분리해 유지보수성을 높였습니다.

Q. 왜 API 로직을 컴포넌트 안에 직접 쓰지 않았나요?

A. API 주소, 헤더, 에러 처리, 로딩 처리를 여러 컴포넌트에 흩뿌리면 관리가 어렵습니다. `services/api.ts`로 모아 변경 지점을 줄였습니다.

Q. 왜 `Product` 타입이 `types`와 `data`에서 같이 언급되나요?

A. 공유 타입은 `src/types/index.ts`에 정의하고, 데이터 파일에서는 해당 타입을 import해 로컬 상품 데이터가 UI 계약에 맞는지 검증합니다.

Q. 왜 `ErrorBoundary` fallback 메시지가 섹션마다 다른가요?

A. 어떤 섹션에서 문제가 생겼는지 사용자가 이해할 수 있게 하기 위해서입니다. 개발자에게도 문제 범위를 좁히는 단서가 됩니다.

Q. 왜 `crypto.randomUUID()`를 쓰나요?

A. 브라우저에서 간단히 충돌 가능성이 낮은 익명 식별자를 만들 수 있기 때문입니다. 단, 인증 토큰으로 쓰면 안 됩니다.

Q. 왜 `localStorage` 초기값을 IIFE로 읽나요?

A. store 생성 시점에 기존 동의 여부와 세션 ID를 확인해 초기 화면에서 바로 동의 상태를 반영하기 위해서입니다.

Q. 왜 `typeof window === "undefined"` 체크가 있나요?

A. 브라우저 환경이 아닐 때 `window` 접근으로 오류가 나는 것을 막기 위한 방어 코드입니다.

Q. 왜 `pointer-events-none`을 쓰나요?

A. 동의 모달 뒤쪽 화면을 클릭하지 못하게 하기 위해서입니다. 시각적 blur뿐 아니라 실제 상호작용도 차단합니다.

Q. 왜 `select-none`을 쓰나요?

A. 동의 전 blur된 배경 콘텐츠가 드래그 선택되는 것을 막아 모달 집중도를 높입니다.

Q. 왜 `scroll-behavior: smooth`를 쓰나요?

A. 다시하기나 네비게이션 이동 시 섹션으로 부드럽게 이동하게 하기 위해서입니다.

Q. 왜 custom scrollbar를 넣었나요?

A. 기본 스크롤바가 브랜드 톤과 다를 수 있어, 전체 UI 톤에 맞게 얇고 절제된 스타일을 적용했습니다.

Q. 왜 `text-rendering: optimizeLegibility`를 쓰나요?

A. 텍스트 렌더링 품질을 높여 제목과 본문 가독성을 개선하기 위한 전역 설정입니다.

Q. 왜 `font-feature-settings`를 쓰나요?

A. 커닝과 ligature 같은 폰트 기능을 활성화해 타이포그래피 품질을 높이기 위한 설정입니다.

## 18. 답변하기 어려운 질문 대응

Q. 이 추천이 실제로 얼마나 정확한가요?

A. 프론트엔드 관점에서는 추천 결과를 사용자가 이해할 수 있게 보여주는 역할을 했습니다. 정확도는 백엔드 모델, 데이터 품질, 평가 지표가 함께 검증해야 하며, 프론트 fallback 점수는 UI 보조 로직입니다.

Q. 개인정보 보호가 완벽한가요?

A. 완벽하다고 말할 수는 없습니다. 프론트엔드는 동의 흐름과 이미지 경량화, 명시적 분석 시작을 구현했고, 실제 보호 수준은 백엔드 저장 정책, 삭제 정책, 접근 제어와 함께 완성되어야 합니다.

Q. 클라이언트에서 파일 검증하면 우회 가능한 것 아닌가요?

A. 맞습니다. 클라이언트 검증은 우회 가능합니다. 그래서 사용자 실수와 기본 공격을 줄이는 1차 방어로 보고, 백엔드에서 반드시 재검증해야 합니다.

Q. 왜 아직 실제 S3 업로드가 아닌가요?

A. 현재 프로젝트 범위에서는 분석 흐름과 UI 경험 검증이 우선이었고, 스토리지 연동은 시뮬레이션으로 분리했습니다. 실제 운영 단계에서는 signed URL 기반 업로드와 보관 정책을 추가해야 합니다.

Q. 접근성 미흡한 부분이 있지 않나요?

A. 있습니다. 일부 카드형 클릭 영역과 모달 focus trap은 보완 대상입니다. 다만 이 부분을 인지하고 있으며, Radix primitive나 native button 전환으로 개선할 계획입니다.

Q. E2E 테스트가 현재 UI 변화에 항상 맞나요?

A. UI 플로우가 바뀌면 E2E 테스트도 같이 업데이트해야 합니다. 특히 이미지 업로드처럼 흐름이 자주 바뀌는 영역은 안정적인 selector와 최신 사용자 흐름 기준으로 유지해야 합니다.

Q. 프론트엔드 fallback 추천이 백엔드 추천과 충돌하지 않나요?

A. 우선순위를 명확히 했습니다. 백엔드 recommendations가 있으면 프론트 fallback은 사용하지 않습니다. fallback은 백엔드 결과가 없을 때 화면 복구용입니다.

Q. TypeScript 타입과 백엔드 실제 응답이 다르면 어떻게 하나요?

A. 현재는 타입 계약과 테스트로 관리합니다. 더 안전하게 하려면 Zod 같은 런타임 검증을 추가해 응답을 받은 즉시 schema 검사를 할 수 있습니다.

Q. 지금 구조가 확장에 충분한가요?

A. 현재 규모에는 충분하지만, 기능이 늘어나면 서버 상태 관리, 접근성 컴포넌트 표준화, API schema 검증, image storage contract 분리가 필요합니다.

## 19. 짧은 마무리 답변 모음

Q. 프론트엔드를 한 문장으로 요약하면?

A. 사용자의 스타일 이미지와 향기 취향을 입력받아 AI 분석 결과를 감성적인 리포트와 추천 UI로 변환하는 React 기반 SPA입니다.

Q. 가장 자신 있는 구현은?

A. 이미지 업로드부터 분석 요청까지의 중복 실행 방지와, 리포트 캡처 품질을 안정화한 부분입니다.

Q. 가장 어려웠던 부분은?

A. 비동기 이미지 처리와 분석 요청이 겹치면서 중복 API 요청이 발생할 수 있는 지점을 막는 것이었습니다.

Q. 가장 많이 배운 점은?

A. 프론트엔드는 화면만 만드는 것이 아니라 데이터 계약, 비동기 흐름, 사용자 대기 경험, 실패 처리까지 함께 설계해야 한다는 점입니다.

Q. 개선한다면 가장 먼저 무엇을 하나요?

A. 실제 스토리지 업로드 계약을 정리하고, 접근성 보완과 E2E 테스트 최신화를 먼저 진행하겠습니다.
